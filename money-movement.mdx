### **Overview**

Direct Pay aggregates individual PX transactions (class P = Payment, type X = Direct Pay) for a creditor payee into a single lump-sum ACH payment, accompanied by an itemized activity report. BetrLink operates this program on behalf of NDR, with Hudson Technology Systems providing the underlying infrastructure.

**End-to-end flow:**

1. NDR sends PX transactions referencing a specific payee.
2. The Daily Swipe Job (2:45 AM) aggregates all PX transactions per payee, creates a Direct Pay Header record, and moves funds internally from individual client sub-accounts to the payee's dedicated sub-account.
3. A lump-sum withdrawal transaction is created from the payee sub-account to Payment Hold.
4. The ACH Out job picks up the withdrawal and submits it to NACHA.
5. An AWS job generates a daily activity report for each active payee, even on days with \$0 in transactions, and delivers it via the payee's configured method.

 

---

 

### **Report Delivery**

Direct Pay reports are delivered via **SFTP** or **email**. The delivery method is set during payee setup and stored on the Salesforce payee record.

**SFTP:** The report is uploaded to the creditor's SFTP server. An email notification is sent to NDR (To field) and CC'd to BetrLink Direct Pay. The creditor does not receive an email.

**Email:** The report is emailed directly to the creditor (To field). NDR is included in the BCC field. The from address displays as BetrLink.

In both cases, a BCC copy can be added to the BetrLink ops inbox for internal quality control. To add an address, edit the BCC field on the payee record in Salesforce.

 

---

 

### **Operational Constraints**

**The Direct Pay Job runs once per day.** It cannot be re-run on the same day unless the Direct Payee explicitly requests a second run. Do not trigger the job manually without that confirmation.

**Do not run the Direct Pay Job when late transactions arrive after 4:00 PM.** In that scenario, run the clear job (ACHInReturnBatch) and the Fee Swipe job instead, then submit ACH Origination. The Direct Pay swipe will pick up those transactions the following day.

 

---

 

### **Add a New Direct Payee**

Adding a new payee is a two-phase process. **Phase 1 (Setup)** is completed by engineering and does not require NDR to be ready. **Phase 2 (Activation)** is a separate step gated on NDR confirming they are ready to send PX transactions. Do not activate a payee before NDR confirms readiness, as an active payee with no incoming transactions will generate empty Direct Pay Header records.

---

 

#### **Phase 1: Engineering Setup**

**Trigger:** BetterLink submits an HSD ticket requesting a new direct payee. The ticket includes the creditor's contact information, preferred delivery method (SFTP or email), and banking details (attached form).

**Ops:**

1. When the HSD arrives, create an engineering task on the engineering board. The engineering task should include:
   - Delivery type (Email or SFTP)
   - Email To, CC, and BCC fields, each appended with `.invalid`
   - Excel template reference and password if applicable
   - A link to the HSD ticket for banking information

**Engineering - Salesforce:**

1. Create the payee record and associated account in Salesforce using the[<u> new direct payee setup script</u>](https://hudson-tech-systems.atlassian.net/wiki/spaces/BET/pages/2696183813/Code+Snips#Steps-to-setup-a-new-Direct-Payee-in-the-system "https://hudson-tech-systems.atlassian.net/wiki/spaces/BET/pages/2696183813/Code+Snips#Steps-to-setup-a-new-Direct-Payee-in-the-system"). Reference the HSD ticket for input values.
2. Set the payee Account Status to **Inactive**.
3. Append `.invalid` to all email addresses on the payee record. This prevents live sends in the event the payee is accidentally activated before it is ready.
4. Once the record is saved, retrieve the auto-generated **API ID** from the payee record.
5. Send the API ID to Weston Bingham (NDR) via the `#direct-pay-setup` Slack channel. NDR requires this ID to configure their system to route PX transactions to the correct payee. NDR can begin their setup immediately after receiving it.

**Engineering - AWS:**

1. Configure report generation for the payee. Apply any report customizations specified in the HSD.
2. Configure the delivery method: SFTP connection or email, per the HSD.
3. Update the relevant Direct Pay delivery fields on the Salesforce payee record.

Once both Salesforce and AWS steps are complete, the engineering ticket can be closed. Ops waits for NDR confirmation before proceeding to Phase 2.

 

---

 

#### **Phase 2: Activation**

**Trigger:** Weston (NDR) confirms via the "direct pay" chat that NDR is ready to send PX transactions for the payee.

**Pre-flight checks — verify all of the following before flipping to Active:**

1. Pull up the original HSD ticket and confirm the contact information on the payee record matches the ticket exactly.
2. Remove `.invalid` from all email addresses on the payee record.
3. **CRITICAL: Verify the routing number is exactly 9 digits.** An incorrect routing number will cause the ACH withdrawal to be rejected, requiring manual intervention to correct. If the number on the record is 8 digits or otherwise suspect, do not proceed. Contact the source listed in the HSD to confirm the correct value.
4. Confirm `Ownership_Type__c = Payee` on the payee account.
5. Confirm `Account_Balance__c ≥ 0` on the payee account.
6. Confirm both routing number and account number fields are populated (complete banking info is required for the swipe to process).

**Activation steps:**

1. Set the payee record Status to **Active**.
2. Navigate to the associated company account and set its Status to **Active**.
3. Send a message to Weston (NDR) in the "direct pay" chat confirming the payee has been activated.

 

---

 

**Post-activation monitoring (morning after activation):**

Check the[<u> Health Monitoring Dashboard</u>](https://hudsonts.lightning.force.com/lightning/r/Dashboard/01Z3h000000IxH5EAK/view?queryScope=userFolders "https://hudsonts.lightning.force.com/lightning/r/Dashboard/01Z3h000000IxH5EAK/view?queryScope=userFolders") and verify the following for the new payee:

- A Direct Pay Header record was created for that day.
- For email-delivery payees: the header status is `Generated` or `Sent`.
- For SFTP-delivery payees: the header status is `Sent`.
- The header has been emailed (applies to all payees, including SFTP).
- The withdrawal amount on the header equals the sum of the associated PX transactions.
- A lump-sum withdrawal transaction was created from the payee sub-account and cleared.

 

---

 

### **Deactivate a Direct Payee**

1. Update `Payee.Direct Pay Report Generation Method` to **None**.
2. Update the Payee Account Status to **Inactive**.

 

---

 

### **Testing**

**Email payees (UAT):** Change the email address on the payee record to an inbox the tester controls.

**SFTP payees:** Test the email delivery path first in UAT to verify the report file looks correct. SFTP connection testing must be done in production:

1. Coordinate with the creditor so they are expecting a blank test file.
2. Set the payee account to **Active**. The Daily Swipe Job will pick it up. Because there are no live PX transactions at this stage, the report file will be blank.
3. Confirm with the creditor that the blank file was received and the connection is working.
4. Set the payee account back to **Inactive** until the DSC is ready to send live PX transactions, unless the creditor has agreed to receive blank files in the interim.

**Known issue — SFTP connection failure on first drop:**

New SFTP payees occasionally fail on their first file drop with a connection error, even when credentials are correct. The most common cause is IP whitelisting that has not yet fully propagated on the creditor's side. If this happens:

- Verify you can authenticate to the creditor's SFTP manually using the credentials in Salesforce.
- If authentication succeeds but the automated drop still failed, have the business email the file to the creditor manually for that day.
- If the issue repeats the following day, escalate to engineering.
- If you cannot authenticate at all, contact the creditor to confirm whitelisting is complete. Refer to the IP addresses in the HSD or the[<u> Direct Payees</u>](https://hudson-tech-systems.atlassian.net/wiki/spaces/BET/pages/3101720583 "https://hudson-tech-systems.atlassian.net/wiki/spaces/BET/pages/3101720583") reference page.
